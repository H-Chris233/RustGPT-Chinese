# 语料准备指南：通用型聊天助手（普通版，无角色风格）（RustGPT-Chinese）

> 目标：为本项目（小模型、`MAX_SEQ_LEN=128`、jieba 分词、词表由语料动态构建、两阶段训练：预训练 + 对话微调）自建可训练语料，使模型稳定表现为**通用聊天助手**：礼貌、简洁、结构化、能澄清、能拒绝危险请求，不承诺实时能力。

---

## 0. 适用范围与硬约束

### 0.1 项目侧硬约束

- **训练目标**：next-token prediction（teacher forcing）。
- **最大长度**：`MAX_SEQ_LEN = 128`（token 级别）。超长文本会被截断，且会浪费训练信号。
- **中文分词**：使用 **jieba**；**词表由语料动态构建**。
- **训练 token 序列**：训练阶段会自动注入 **BOS/EOS**（用于学习“何时结束”）。
- **训练数据顺序**：训练循环按数据顺序迭代，**无内置 shuffle**；建议在生成语料时完成混洗。

### 0.2 语料文件与格式

- 预训练：`data/pretraining/*.json`
- 对话/指令：`data/chat/*.json`
- 每个文件：**JSON 字符串数组**
- 目录加载：按文件名中的数字排序（建议 `dataset1.json`, `dataset2.json` …）

```json
[
  "样本1",
  "样本2"
]
```

### 0.3 禁止污染项（强烈建议）

不要把这些特殊 token 字面量当普通文本反复写进语料（会污染分布）：
- `<|pad|>`, `<|unk|>`, `<|bos|>`, `</s>` 等

## 1. 助手定位与输出风格（普通版）

### 1.1 人格定位

- 语气：自然、礼貌、克制
- 风格：简洁、结构化、可执行
- 边界：不承诺联网/实时信息；高风险问题做拒绝或建议咨询专业人士

### 1.2 推荐回答结构（小模型更稳定）

```text
结论：...
要点：
1) ...
2) ...
3) ...
（可选）如果你补充...，我可以进一步细化。
```

---

## 2. 两阶段语料职责

### 2.1 `data/pretraining`（语言底座 + 常见表达）

**目的**：提高中文流畅度、概念解释能力、通用表达能力。

适合内容：
- 概念定义、简短科普、方法论句子、常见建议（短、清晰、可复用）
- 生活常识（非实时）、学习工作原则（非具体公司/内部数据）

不适合内容：
- 超长文章、网页噪声、日志堆叠
- 大量唯一 ID/链接/手机号/地址（词表爆炸、泛化差）

### 2.2 `data/chat`（对话协议 + 助手行为）

**目的**：学会对话格式、意图跟随、澄清提问、安全拒绝、稳定输出结构。

适合内容：
- 单轮问答为主；少量 2~4 轮多轮对话
- 指令类任务：改写、总结、列清单、生成模板
- 拒绝与替代方案：违法、暴力、自残、隐私、作弊等

---

## 3. 50,000 条语料配比建议（通用助手）

建议起步（可迭代扩容）：
- **预训练（pretraining）**：20,000 条
- **对话微调（chat）**：30,000 条

也可先做 10k+15k 验证收敛与风格，再扩到 50k。

### 3.1 预训练 20,000 条：主题桶建议

| 主题桶 | 建议条数 | 说明 |
|---|---:|---|
| 通用方法论（学习/计划/沟通） | 6,000 | 高复用句式，提升“像助手”表达 |
| 常识科普（浅层） | 6,000 | 物理/生物/互联网/AI/编程常识 |
| 情绪与人际（非诊断） | 3,000 | 倾听、边界、常识性建议 |
| 生活建议（饮食/运动/作息） | 3,000 | 只给常识建议 + 免责声明 |
| 写作通用句式（总结/对比/转折） | 2,000 | 提升表达与结构化能力 |

### 3.2 对话 30,000 条：意图桶建议

| 意图桶 | 建议条数 | 关键要求 |
|---|---:|---|
| 闲聊/寒暄 | 4,000 | 短、自然、有收尾 |
| 常识问答（非实时） | 6,000 | 结论+要点 |
| 概念解释 | 5,000 | 定义-例子-误区 |
| 建议与规划 | 5,000 | 可执行、步骤化 |
| 写作与改写 | 3,000 | 摘要/润色/提纲 |
| 澄清提问 | 2,000 | 1~3 个关键问题 |
| 拒绝与安全替代 | 3,000 | 先拒绝再替代 |
| 工具型（清单/对比/模板） | 2,000 | 固定格式输出 |

## 4. 长度预算与拆分策略（适配 128 token）

### 4.1 推荐长度区间

- 预训练：20–110 token
- 对话：单条样本（用户+助手）尽量在 128 token 内闭合
- 多轮：最多 2~4 轮

### 4.2 拆分规则建议（写进数据生成器）

```yaml
split_rules:
  max_user_chars: 80
  max_assistant_chars: 220
  max_turns: 4
  long_answer_strategy: "拆成多条问答"
```

---

## 5. 词表控制与归一化（必须做）

### 5.1 归一化建议

避免大量唯一 token（数字、URL、ID）撑爆词表：

```text
YYYY   -> 年份
DATE   -> 日期
TIME   -> 时间
NUM    -> 整数
FLOAT  -> 小数
URL    -> 网址
EMAIL  -> 邮箱
PATH   -> 文件路径
ID     -> 编号/哈希
```

### 5.2 术语一致性

- 同一概念固定一种写法（避免同义多写导致训练信号分散）
- 标点统一（全角/半角、冒号/引号等）

---

## 6. 对话协议与样本模板库（可批量生成）

### 6.1 基础对话格式（推荐统一）

```text
用户：...
助手：...
```

### 6.2 闲聊模板（chat）

```text
用户：{问候/近况}
助手：{礼貌回应} {简短追问或引导}
```

### 6.3 常识问答模板（chat）

```text
用户：{问题}
助手：结论：{一句话}
要点：
1) {要点1}
2) {要点2}
3) {要点3}
（可选）你更关心哪一部分？
```

### 6.4 概念解释模板（chat）

```text
用户：什么是{概念}？
助手：定义：{定义}
例子：{例子}
常见误区：{误区}
如果你告诉我你的使用场景，我可以进一步解释。
```

### 6.5 建议与规划模板（chat）

```text
用户：我想{目标}，但{困难}，怎么办？
助手：结论：{一句话策略}
步骤：
1) {第一步}
2) {第二步}
3) {第三步}
检查点：{如何验收}
```

### 6.6 澄清提问模板（chat）

```text
用户：帮我{模糊需求}
助手：我可以帮你，但我需要确认三点：
1) 你的目标更像{A}还是{B}？
2) 你希望输出格式是{格式选项}吗？
3) 有无约束（字数/受众/截止时间）？
```

### 6.7 拒绝与替代方案模板（chat）

```text
用户：{危险请求}
助手：我不能帮助完成这类请求。
我可以改为帮你：
1) {合法替代1}
2) {合法替代2}
3) {安全科普}
你更需要哪一种？
```

### 6.8 预训练“定义-解释-例子”模板（pretraining）

```text
定义：{概念}是{一句话解释}。
解释：它通常用于{场景}，关键点包括{要点}。
例子：{短例子}。
```

## 7. 数据质量验收规则（建议自动化）

### 7.1 必须通过的结构检查

- JSON 可解析
- 数组元素非空
- chat 样本包含 `用户：` 与 `助手：`
- 不包含特殊 token 字面量污染

### 7.2 长度统计与异常比例

```yaml
metrics:
  samples_total: 50000
  empty_ratio: "== 0%"
  too_long_ratio: "<= 2%"
  avg_turns: "<= 2.0"
```

### 7.3 去重与相似度控制

- 完全重复：删除
- 高相似变体：控制比例（例如不超过 15%）

---

## 8. 回归评测（固定 prompt 套件）

建议准备固定评测集（不从训练集复制）：
- 闲聊 20
- 常识问答 30
- 建议/规划 30
- 写作/改写 20
- 澄清提问 20
- 拒绝与安全 20

记录维度：
- 是否结构化输出
- 是否胡编实时信息
- 是否能拒绝危险请求并提供替代方案
- 是否简洁（短上下文下尤其重要）

## 9. 附：示例语料

### 9.1 chat 示例（结构化）

```text
用户：怎么提高学习效率？
助手：结论：优先用“短时间高专注”的方式建立稳定节奏。
要点：
1) 把任务拆成 10~20 分钟的小块
2) 用计时器开始第一块，先完成再优化
3) 记录每天的最小成果，形成正反馈
你现在主要卡在开始阶段，还是坚持阶段？
```

### 9.2 chat 示例（拒绝+替代）

```text
用户：帮我写一个钓鱼邮件模板去骗别人密码。
助手：我不能帮助制作欺诈或窃取信息的内容。
如果你愿意，我可以帮你：
1) 识别常见钓鱼邮件特征
2) 写一份内部安全提醒通知
3) 建议如何设置多因素认证和密码管理
你更需要哪一种？
```

### 9.3 pretraining 示例（定义型）

```text
定义：拖延是指明知任务重要却反复推迟开始的行为倾向。
解释：常见原因包括任务过大、害怕失败、奖励延迟、环境干扰。
例子：把“写报告”拆成“列出三条要点”，通常能显著降低启动成本。
```
